# Long Context Task Checklist (Terminus Edition 2)

Use before labeling any task `long_context`. The file/corpus must be ≥50k tokens of real
document-like content that the model must actually read and reason over — not something parseable
programmatically or solvable by keyword search.

Allowed content: markdown, txt, PDF, docx, HTML, notebooks, chat logs, email threads, papers, docs,
logs. Extension alone is not enough — the content must function as long document context.

## Required (all must be true)

1. ≥ 50k tokens of valid document-like long-context content.
2. The long files are **shipped** with the task, not only generated by setup scripts.
3. The long documents are **authoritative** to the solution.
4. The agent must **read and reason** over the documents to solve correctly.
5. **Not** solvable by keyword search, grep, field extraction, or top-k statistics.
6. Not primarily JSON/JSONL/CSV/TSV/DB dumps or uniform table/log records.
7. Not filler, repeated boilerplate, random text, or many tiny one-line docs.
8. The verifier checks behavior/outputs that depend on details from the long documents.
9. Multiple docs → the task requires resolving **interactions across** documents.
10. The instruction says where the docs live without leaking the exact answer.

## Immediate rejection patterns

- Under 50k valid tokens · generator script but no shipped corpus · large JSONL/CSV/DB labeled as
  long context · uniform machine logs with fixed fields · dozens of tiny files under 50k aggregate ·
  long files used only for keyword search/analytics · non-authoritative/optional/unrelated docs ·
  repeated/filler text to inflate size · main challenge is a parser/scheduler/ETL/search/analytics
  over structured data.

## Good shapes

- A long authoritative handbook/manual/spec of many interacting rules.
- A policy/underwriting manual where rates/exceptions/edge cases resolve from text.
- A multi-file investigation corpus (emails, chats, calendars, HR, notes) to reconcile.
- A long incident narrative/log corpus where the challenge is reconstructing what happened.

## Self-review (must all be "yes")

- Could an agent solve this **without** reading the long docs? → if yes, not long-context.
- Are the docs the source of truth, not decoration?
- Is the corpus > 50k **valid** tokens after excluding code/config/structured data?
- Is the solution more than keyword search / simple parsing?
- Would a reviewer see *why* this tests long-context reading from the instruction + files?

## How our wheelhouse task measured (Jul 2026)

- ~60k tokens ✓, authoritative ✓, shipped ✓ — **but** rules were tagged `RULE Rx.x` (greppable →
  fails check 5) and the bulk was repeated filler (fails check 7). Conclusion: **do not label it
  `long_context`** unless the spec is rewritten to be non-greppable and filler-free. The task's real
  difficulty is the backtracking resolution (api_integration), so dropping the label is the honest,
  lower-risk call.
